# Claude 지침

이 지침은 자주 일어나는 LLM 코딩 실수를 줄이기 위한 행동 규약이다. 사소한 작업에는 판단력으로 보완하되, 기본은 속도보다 신중함에 무게를 둔다.

## 응답 언어
- 항상 한국어로만 응답한다.
- 사용자에게 보여지는 모든 응답은 존댓말(합쇼체/해요체)로 작성한다. 반말("~한다", "~해줘", "~알려달라" 등)은 금지한다.

## 사고 방침 (구현 전)
- 가정을 명시하고, 불확실하면 질문한다.
- 여러 해석이 가능하면 모두 제시하고 임의로 고르지 않는다.
- 더 단순한 접근이 있으면 말한다. 필요하면 반박한다.
- 불명확한 것은 멈추고 무엇이 모호한지 짚어 묻는다.
- 혼란을 숨기지 않고 트레이드오프를 표면화한다.

## 단순함 우선
- 문제를 푸는 최소 코드만 작성. 투기적 구현 금지.
- 요청되지 않은 기능 추가 금지.
- 1회용 코드에 대한 추상화 금지.
- 요청되지 않은 "유연성" 이나 "설정 가능성" 금지.
- 일어날 수 없는 상황에 대한 에러 처리 금지.
- 200 줄을 50 줄로 줄일 수 있다면 다시 쓴다.
- 시니어 엔지니어가 "과설계" 라고 할 만한지 자문한다.

## 구현 품질 방침
- 컴포넌트나 기능을 구현할 때 실제 사용 가능한 수준의 스펙을 갖추도록 고민한다.
- 경계 조건(아이템 0개, 크기 불균일, 가변 크기 등)을 고려한 로직을 작성한다.
- 특수 케이스가 아닌 일반 케이스를 기본으로 설계하고, 특수 동작은 setter 로 제공한다.
- 내부 계산은 외부 하드코딩 상수에 의존하지 않고 런타임에 실제 데이터를 읽어 처리한다.

## 외과적 수정 (Surgical Changes)
- 가지의 정합성을 맞추기 위해 뿌리 부분의 정합성을 훼손하지 않는다. 파생 클래스나 상위 계층의 문제를 해결하려고 기반 클래스나 하위 계층의 설계를 변경해서는 안 된다.
- 기존 코드의 표기법(네이밍 컨벤션, 들여쓰기, 주석 스타일 등)을 그대로 따른다.
- 코딩 요청 시 필요한 부분만 최소한으로 수정한다. 불필요한 리팩토링, 추가 기능, 스타일 변경을 하지 않는다.
- 기존 코드를 수정할 때 인접한 코드 / 주석 / 포맷을 "개선" 하지 않는다.
- 망가지지 않은 것을 리팩터하지 않는다.
- 본인 스타일이 다르더라도 기존 스타일을 따른다.
- 무관한 데드 코드를 발견하면 언급만 하고 삭제하지 않는다.
- 본인 변경으로 미사용이 된 import / 변수 / 함수만 제거한다. 기존 데드 코드는 요청 없이 제거하지 않는다.
- 모든 변경 라인은 사용자의 요청에 직결되어야 한다.

## 코딩 방침
- `libs/vanilla.js` 에 있는 기존의 코딩 스타일을 무조건 따른다.
- 새로운 게임은 반드시 `main.js` 또는 메인 내비게이션에 통합되도록 한다.
- `assets/` 안의 자산 구조를 일관되게 유지한다.
- 프로젝트 내에 존재하지 않는 가상의 함수나 클래스를 창작하여 사용하지 않는다. 반드시 제공된 코드베이스 내의 요소만 사용한다.

## 필수 코딩 스타일
- **변수명 유지 및 축약 금지:** 기존 코드의 변수명을 임의로 변경하지 않는다. 변수명을 줄여 쓰거나 약어를 사용하는 것을 금지한다. (예: `canvasRenderingContext` 를 `ctx` 로 축약 금지, `button` 을 `btn`, `value` 를 `val`, `duplicate` 를 `dup` 으로 축약 금지) 이 규칙은 변수명뿐 아니라 UI 레이블, 버튼 텍스트 등 화면에 표시되는 모든 문자열에도 동일하게 적용한다.
- **지역 변수 할당 강제:** `get` 메서드나 프로퍼티의 반환값을 다른 함수의 인자로 직접 전달하지 않는다. 반드시 풀네임의 지역 변수에 먼저 할당한 후 사용한다.
- **다중 구문 작성 금지:** 한 줄에 세미콜론(`;`)을 하나 초과하여 사용하지 않는다. 각 구문은 반드시 새로운 줄에 작성한다.
- **중괄호 강제:** 모든 조건문, 반복문, 함수는 반드시 중괄호 `{}` 를 사용한다. 1줄이라도 생략을 금지한다. 여는 중괄호 `{` 는 시작행의 끝에, 닫는 중괄호 `}` 는 새로운 행에 위치해야 한다.
- **조건문 개행 강제:** `else`, `else if`, `catch`, `finally` 등은 반드시 다음 행에서 시작한다. (예: `}\nelse {`)
- **switch 문 중괄호 강제:** `switch` 문의 모든 `case` 와 `default` 블록은 반드시 중괄호 `{}` 를 사용한다. `break;` 는 반드시 해당 블록 내부에 위치해야 한다.
- **Private 멤버 변수:** 모든 클래스의 `private` 멤버 변수는 반드시 `#` 접두사를 사용한다. 단, `private` 메서드(`#` 접두사 함수)는 사용하지 않는다. 메서드는 항상 공개(public) 또는 일반 메서드로 작성한다.
- **Private 변수 접근 강제:** 프라이빗 변수(`#` 접두사)의 값이 필요할 때는 반드시 해당 변수를 반환하는 getter 메서드를 통해서만 접근한다. 프라이빗 변수를 직접 참조하지 않는다.
- **주석 및 스타일 보존:** JSDoc, 함수 헤더 주석(`//=================`), 인라인 주석 등 모든 주석과 기존 코딩 스타일을 삭제하거나 임의로 수정하지 않는다.
- **변경 이력 주석 금지:** 코드를 수정할 때 "무엇을 변경했다" 는 식의 불필요한 변경 이력 주석을 코드 내에 추가하지 않는다.
- **전역 내장 객체 접근 강제:** `Object`, `Promise`, `Math`, `JSON`, `Array` 등 전역 내장 객체 및 함수를 사용할 때는 반드시 `System.` 접두사를 붙인다. (예: `Object.freeze(...)` → `System.Object.freeze(...)`) 파일 상단에 `const System = globalThis;` 선언이 있는 경우에 적용한다.
- **문자열 따옴표 통일:** 모든 코드에서 문자열은 쌍따옴표(`"`) 또는 템플릿 리터럴(백틱 `` ` ``)만 사용한다. 홑따옴표(`'`)는 사용하지 않는다.

## 목표 주도 실행
- 작업을 검증 가능한 목표로 변환한다.
  - "검증 추가" → "유효하지 않은 입력 케이스 테스트 작성 후 통과시킴"
  - "버그 수정" → "버그를 재현하는 테스트 작성 후 통과시킴"
  - "X 리팩토링" → "리팩토링 전후 테스트 통과 확인"
- 다단계 작업의 경우 짧은 계획을 명시한다.
  ```
  1. [단계] → 검증: [확인 방법]
  2. [단계] → 검증: [확인 방법]
  3. [단계] → 검증: [확인 방법]
  ```
- 강한 성공 기준은 독립적인 반복을 가능하게 한다. "되게 만들어" 같은 약한 기준은 끊임없는 명확화 요청을 부른다.

## 빌드 검증 방침
- 코드 수정 후 빌드 검증(esbuild 등)을 실행하지 않는다.

## UI 배치 방침
- UI 배치 작업은 무조건 NodeLayout 빌더 패턴으로 작성한다.

## 작업 로그 방침
- `.logs/claude-YYYY-MM-DD.log` 형식으로 `.logs` 디렉토리에 날짜당 하나의 로그 파일만 유지한다. 동일 날짜 파일이 이미 존재하면 이어서 기록한다.
- 각 작업 항목에 요청 타임스탬프와 응답 타임스탬프를 `[HH:MM 요청 → HH:MM 응답]` 형식으로 기록한다.
- 번호 붙은 분할 로그(`-2`, `-3` 등)는 만들지 않는다.

---

이 지침이 잘 작동하고 있다면: diff 에 불필요한 변경이 줄어들고, 과설계로 인한 재작성이 줄어들며, 명확화 질문이 실수 이후가 아니라 구현 이전에 나온다.
